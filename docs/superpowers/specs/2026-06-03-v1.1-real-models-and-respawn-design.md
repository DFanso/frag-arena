# Design: v1.1 — Real 3D Models + Respawn Enhancement

**Date:** 2026-06-03
**Status:** Approved (pending written-spec review)
**Owner:** DFanso
**Builds on:** the shipped v1 FPS (`docs/superpowers/specs/2026-06-03-cloudflare-multiplayer-fps-design.md`)

## 1. Overview

Two enhancements to the working v1 FPS, on branch `build/real-models`:

1. **Real 3D models (CC0):** replace the procedural box visuals with real, free (CC0)
   glTF models — animated player characters, a first-person weapon viewmodel, arena props
   (crates/barrels), and a gradient sky with proper lighting/shadows.
2. **Respawn enhancement:** smart spawn placement (respawn at the spawn point farthest from
   the nearest living enemy) plus a death screen with a respawn countdown.

**Hard guarantee:** the network protocol, the Durable Object game logic, and the capsule
collider are NOT changed by the visual work. Collision stays box-based and decoupled from
visuals. The only server change is the spawn-point *selection* logic (Part B).

## 2. Goals / Non-goals

**Goals:** animated characters driven by existing netcode; FP weapon + muzzle flash; nicer
environment (props + sky + shadows); smart respawn placement; death/respawn countdown UI;
everything still 100% free and self-contained (assets bundled, served as free static assets).

**Non-goals (YAGNI):** no new weapons/maps/skins; no protocol changes; no server-side physics;
no ragdolls/death animations beyond a simple cue; no model LODs; no character customization.

## 3. Asset strategy

**Bundle CC0 GLBs in the repo** (`public/models/`), loaded at runtime via Three.js
`GLTFLoader` from the free static-assets path. Self-contained, offline, no runtime CDN
dependency. Every model load has a **fallback** so the game still runs if an asset is missing
(character → the current box body; gun → a primitive-built gun; props → colored boxes).

- **Character:** `RobotExpressive.glb` — CC0 (Tomás Laulhé, modified by Don McCurdy). Ships
  with `Idle`, `Walking`, `Running`, `Jump`, `Punch`, `Wave`, `Death`, etc. Known-good CC0 file.
- **Weapon, props (crate/barrel):** verified CC0 sources (Kenney / Quaternius / Poly Pizza)
  whose exact download URLs are **locked and verified during the implementation plan**, with
  the primitive/box fallbacks above. Asset provenance + licenses recorded in `public/models/CREDITS.md`.
- **Sky:** procedural gradient (shader/vertex-color sphere) — no asset.

All GLBs are small (tens–hundreds of KB) → far under the 25 MiB/file and 20,000-file static
asset limits, and they live outside the JS bundle (loaded as separate assets).

## 4. Part A — Real 3D models

### 4.1 Asset loader (`src/assets.ts`)
- `loadAssets(): Promise<AssetRegistry>` — preloads every GLB with `GLTFLoader`, resolves to a
  typed registry `{ character: GLTF | null, gun: GLTF | null, crate: GLTF | null, ... }`.
- Shows a "Loading…" overlay (reuses the HUD overlay container) until resolved.
- On any individual load error: logs a warning, leaves that entry `null` (consumer uses fallback),
  and never rejects the whole load. `main.ts` `await`s this before starting the game loop.

### 4.2 Animated players (`RemotePlayer` refactor in `src/player.ts`)
- Each `RemotePlayer` gets a `SkeletonUtils.clone()` of `registry.character` (a skinned mesh can't
  be shared raw) + its own `THREE.AnimationMixer`; clips indexed by name.
- **Animation selection** is a pure helper `pickAnim(speedXZ, shootingUntilMs, nowMs)` in a new
  three-free module `src/anim.ts` → `{ base: "Idle" | "Running", shootOverlay: boolean }`. Driven by
  the **velocity already in each snapshot** (`PlayerSnap.v`): `|v.xz| < ε` → Idle else Running.
- `update(nowMs)` advances the mixer (`mixer.update(dt)`), positions/rotates the cloned model
  (yaw only) from the interpolated snapshot, and cross-fades to the selected base clip.
- **Shoot cue:** `RemotePlayer.playShoot()` plays the `Punch` (or `Wave`) clip once as a recoil
  tell. `main.ts` calls it when a `hit`/`kill` event arrives with `by === this remote's id`
  (no protocol change; misses simply don't animate).
- **Hit detection unchanged:** the visual model is flagged `userData.noHit`; an **invisible box
  proxy** (`0.7×1.0×0.7`, the current body size) tagged `userData.playerId` is the raycast target.
  `combat.ts` keeps working as-is.
- If `registry.character` is `null`, fall back to the current colored box body.

### 4.3 First-person weapon viewmodel (`src/viewmodel.ts`)
- `class Viewmodel` — adds the gun model (or a primitive-built gun fallback) as a child of the
  camera, positioned bottom-right of the view; methods `recoil()` (quick kick-back + return via a
  small spring/lerp each frame) and `flash()` (brief additive muzzle-flash sprite/point-light).
- `main.ts` calls `recoil()` + `flash()` from the existing `wireShooting` `onLocalShoot` callback
  (already firing on local shots), alongside the existing shoot SFX.
- `update(dt)` eases the recoil offset back to rest each frame.

### 4.4 Arena: decouple collision from visuals (`src/map.ts` refactor)
- `buildArena()` returns `{ collision: THREE.Group, visual: THREE.Group }`.
  - `collision` = the existing boxes (ground/walls/cover/platform/ramps) — **unchanged geometry**;
    `physics.buildOctree(collision)` uses exactly this, so movement is identical. Not added to the scene (invisible).
  - `visual` = textured/colored ground + walls, plus CC0 **crate/barrel prop models placed at the
    four cover-box positions** and on the platform. Props are `userData.noHit` and never feed the Octree.
  - If a prop model is `null`, the visual group falls back to the colored cover boxes (current look).
- `main.ts` adds `visual` to the scene and builds the Octree from `collision`.

### 4.5 Sky + lighting (`main.ts` scene setup)
- Gradient sky (large back-side sphere with a vertical color gradient, or `scene.background` gradient).
- Lighting: `HemisphereLight` (sky/ground fill) + a `DirectionalLight` with shadow mapping enabled;
  renderer `shadowMap.enabled = true`; ground/props/characters cast+receive shadows. Tuned so the
  low-poly models read clearly.

## 5. Part B — Respawn enhancement

### 5.1 Smart spawn placement (server — `worker/room.ts` + pure helper in `worker/validate.ts`)
- New pure helper `chooseSpawn(spawnPoints: Vec3[], enemies: Vec3[], rand: () => number): Vec3`:
  - No living enemies → a `rand()`-chosen spawn point.
  - Else → the spawn point maximizing distance to the **nearest** living enemy; ties broken with `rand()`.
- `worker/room.ts`: both initial spawn (`addPlayer`) and respawn (`spawn`) call
  `chooseSpawn(SPAWN_POINTS, livingEnemyPositions(excludingSelf), Math.random)`. This **replaces**
  the deterministic `pickSpawn(id)`. (`Math.random` is available in the Workers runtime.)
- Unit-tested via the pure helper with a seeded/stub `rand` (deterministic): enemy clustering →
  asserts the farthest point is chosen; no-enemies → asserts a valid point is returned.

### 5.2 Death screen + countdown (client — `src/hud.ts` + `src/main.ts`)
- On the local player's death (a `kill` event with `on === myId`, or local `hp` reaching 0 in a
  snapshot), `hud.showDeath(killerName, respawnMs)` displays an overlay: **"Fragged by <name>"**
  + a live **3 → 2 → 1** countdown driven off `RESPAWN_MS` (from `protocol.ts`).
- Killer name resolved from the latest snapshot players (by `kill.by` id); unknown → "the void".
- On the local player's `spawn` (`SpawnMsg` with `id === myId`), `hud.hideDeath()` clears the overlay
  (the existing `controls.setPosition(p)` already repositions the player).
- The pure helper `countdownText(remainingMs): string` lives in a new three/DOM-free module
  `src/death-ui.ts` and is unit-tested; `hud.ts` imports it and does the DOM rendering (manual/build
  verified). While dead, local input is ignored for combat by the server already; the overlay is
  `pointer-events:none`.

## 6. Protocol / netcode impact

**None.** `PlayerSnap.v` (velocity) already exists → drives animation. `KillMsg.by` already exists →
drives killer name + remote shoot cue. `SpawnMsg` already exists → drives death-overlay clear +
reposition. No message shapes change; the DO tick/validation are untouched except spawn *selection*.

## 7. Testing

- **Unit (workers vitest pool, three-free modules only):** `pickAnim` (velocity/shoot → clip),
  `chooseSpawn` (smart placement, deterministic via stub rand), `countdownText`.
- **Server:** extend `room.test.ts` to assert respawn lands on a valid spawn point and that, with a
  clustered enemy, the chosen point is the far one.
- **Build/manual:** `tsc --noEmit` + `vite build`; then `npm run dev`, multi-tab — confirm animated
  characters (idle/run), FP gun + muzzle flash + recoil, props/sky/shadows, smart respawn distance,
  and the death-screen countdown.

## 8. File map

- New: `src/assets.ts`, `src/anim.ts`, `src/viewmodel.ts`, `src/death-ui.ts` (three/DOM-free),
  `public/models/*.glb`, `public/models/CREDITS.md`, and tests `test/anim.test.ts`,
  `test/spawn.test.ts`, `test/death-ui.test.ts`.
- Changed: `src/player.ts` (model + mixer + hit-proxy), `src/map.ts` (`{collision, visual}`),
  `src/main.ts` (await assets, mixers, sky/lights, viewmodel wiring, death-screen wiring),
  `src/hud.ts` (death overlay), `worker/validate.ts` (`chooseSpawn`), `worker/room.ts` (use chooseSpawn).

## 9. Risks

- **Exact CC0 gun/prop URLs** — locked + downloaded + verified in the plan; primitive/box fallbacks
  guarantee the game runs regardless. Character (RobotExpressive) is a known-good CC0 file.
- **Skinned-mesh cloning** — must use `SkeletonUtils.clone`, not `.clone()`, or animations break.
- **Perf with N animated characters** — fine for the ~10-player cap; mixers are cheap at low-poly.
- **Shadow cost on low-end GPUs** — use a single directional shadow with a modest map size.
